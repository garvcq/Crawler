# Daily Design Journal · Day 15

## Section 1 – Specific Bug

Today's focus was evaluating optimization strategies for unique URL state management and mapping the control flow of the Web Crawler Coordinator.

During our code review, page deduplication emerged as a potential bottleneck. Querying the MySQL database for every newly discovered link to verify if it has already been parsed would introduce significant database transaction overhead and elevate crawl page latency. Furthermore, we needed a structured way to visualize the entire crawler loop, resume states (`clear`, `keep`, `resume`), and page-rendering criteria.

---

## Section 2 – Failed Attempt

Initially, I considered removing the in-memory `SeenStore` FNV-1a hash map altogether to simplify the architecture, relying entirely on the relational database fields. However, load-testing showed that database latency would degrade crawling speed exponentially at higher depth limits. Consequently, we confirmed that retaining the memory-based `SeenStore` cache is critical for performance. 

For the control flowchart, my first draft missed capturing the conditional Chrome DevTools Protocol (CDP) fallback heuristics. After identifying these gaps, I constructed detail boxes mapping the `needsRendering` criteria (e.g. empty HTML, minimal page size in bytes, or specific SPA framework containers like `#root`).

## Section 4 – Learning Reflection

Today highlighted that caching layers like `SeenStore` are vital when integrating high-concurrency loops with relational database engines. Eliminating memory lookups in favor of database transactions seems simpler but acts as a performance trap. 

Additionally, visual flowchart modeling forces you to document and think through every edge case—such as what triggers a CDP headless render vs a raw HTTP download. By the end of the day, our code review was completed, the control flow graphs were fully detailed, and the system design videos were recorded.
