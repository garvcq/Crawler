# Daily Design Journal · Day 17

## Section 1 – Specific Bug

Today's focus was assembling the background Indexer daemon execution loop, managing transactional write performance, and building a relational search frequency retrieval CLI.

As we ran initial crawl indexes, we hit a severe performance bottleneck: inserting index occurrences row-by-row for thousands of unique words on a single page took minutes to finish. Additionally, our first CLI search queries only retrieved raw page IDs from the occurrence index, leaving us without the actual page URLs or page titles.

---

## Section 2 – Failed Attempt

To solve the database insert bottleneck, I initially attempted to optimize specific MySQL index buffers. This proved ineffective because the overhead of separate network/commit requests for every word was the real bottleneck. I resolved this by wrapping the entire page indexing sequence inside a database transaction block (`START TRANSACTION` and `COMMIT`), reducing write times to milliseconds.

For search, I initially ran standalone queries against `word_occurrences` and then followed with individual select lookups for titles. This nested loop structure multiplied queries. I replaced this with an optimized, single relational database query utilizing `JOIN` links across `word_occurrences`, `words`, `pages`, and `indexed_pages` tables.

## Section 4 – Learning Reflection

Today proved that transactional batching is a mandatory requirement, not an optional optimization, for high-frequency database insertions. Relational databases carry massive disk-commit overhead; minimizing these commits is the single biggest performance gain. 

We also learned the power of SQL JOINs over client-side query chains. Moving lookup logic directly to the database engine drastically cuts back and forth latency. By the end of the day, both the indexer and the search console binaries compiled cleanly, database inserts completed in milliseconds, and the search engine CLI successfully displayed ranked titles and URLs.
